# Desktop 日志恢复、规则管理与设置页设计

**日期**：2026-05-21  
**状态**：待用户审阅  
**范围**：`desktop` Tauri/Vue 客户端 + `agent` 设置与日志保留 API

---

## 背景

Tauri/Vue 版桌面端已经支持项目标签页和跨服务日志搜索。现在还有几处体验没有收口：

1. 应用重启后打开某个服务日志面板，面板只等待实时 WebSocket，缺少最近上下文。
2. 日志面板仍保留旧的 run 历史入口，但现在跨服务搜索已经是历史排障的主要入口。
3. 过滤规则管理入口还是小号占位按钮，规则新增、编辑、保存临时过滤条件的体验不完整。
4. Tauri 版还没有设置页，Swift 版 `SettingsView` 中的通用和项目管理能力需要迁移过来；MCP 暂不处理。

---

## 目标

1. 打开服务日志面板时自动显示最近历史日志，并用分隔线标出历史消息和实时输出的边界。
2. 从日志面板移除旧 run 历史记录入口，历史排障统一使用项目搜索标签页。
3. 补齐项目级日志过滤规则管理，包括新增、编辑、删除、启用/禁用，以及把当前临时 chip 保存为规则。
4. 新增设置页，包含通用设置和项目设置。
5. 通用设置支持日志保留天数和 SuperDev 桌面应用随系统登录启动。
6. 项目设置支持添加/删除项目、服务显示隐藏、启动选中服务配置。

---

## 非目标

1. 不改造跨服务搜索页的信息架构。
2. 不在日志面板中继续提供 run 选择器；`GET /api/logs` 的 run 查询能力后端保留。
3. 不做 MCP 设置页。
4. 不让 agent 脱离桌面应用独立开机常驻；开机自启只控制 SuperDev 桌面应用。
5. 不把服务显示/隐藏写入项目 `.superdev/config.yaml`，它是桌面端本地 UI 偏好。

## 日志面板恢复

### 行为

`LogPanel` 订阅某个 `serviceId` 时，`logStore` 先通过现有接口拉取最近历史日志：

```http
GET /api/logs?service={serviceId}&limit=200
```

历史日志写入该服务的实时 buffer，随后 WebSocket 继续追加实时日志。这样应用重启后，用户打开日志面板也能立即看到最近上下文。

如果历史日志非空，面板在历史日志末尾插入一条展示分隔线：

```text
──── 历史消息 · 之后为实时输出 ────
```

分隔线是展示项，不是 `LogEntry`，不参与复制、导出或后端持久化。过滤规则仍作用于真实日志；分隔线跟随历史/实时边界显示。

如果历史为空，不展示分隔线。

### 去除旧历史入口

日志面板工具栏移除旧的“历史记录”按钮和 run 选择入口。`LogPanel` 不再维护 `viewingRunId` 作为用户可切换模式。历史排障统一从项目搜索标签页进入。

后端 `GET /api/logs` 的 `run` 参数保留，避免影响已有测试和未来兼容。

## 过滤规则管理

### 工具栏入口

日志面板工具栏保留齿轮 icon 作为规则管理入口，但按钮尺寸需要清晰可点，并提供 tooltip。按钮绑定当前 `projectId`；没有项目上下文时禁用。

### 规则管理弹层

点击齿轮后打开项目级规则管理弹层。弹层职责：

- 列出现有 `LogRule`：名称、类型、关键词、AND/OR、启用状态。
- 新增规则：填写名称、包含/排除、关键词列表、AND/OR、默认启用。
- 编辑规则。
- 删除规则。
- 启用/禁用规则。

所有保存操作统一走现有后端接口：

```http
GET /api/projects/{id}/rules
PUT /api/projects/{id}/rules
```

规则持久化到对应项目的 `.superdev/config.yaml`。

### 快捷规则 chip

日志面板过滤栏中已有的项目规则小 chip 保留。它们只作为快速启用/禁用入口：点击 chip 切换 `enabled` 并保存规则列表，不进入编辑表单。

### 临时 chip 保存为规则

当前面板的临时过滤 chip 可以一键保存成项目级规则。入口放在规则管理弹层内，弹层打开时可读取当前面板 chip：

1. 用户点击“从当前过滤保存”。
2. 表单预填当前 chip 的关键词、类型和 AND/OR。
3. 用户确认规则名、类型、关键词和逻辑。
4. 保存走 `PUT /api/projects/{id}/rules`，写入项目配置文件。

## 设置页

### 入口

设置页是 Tauri/Vue 主窗口内的页面或视图：

- 托盘右键菜单“设置…”显示主窗口并跳转设置页。
- 主界面侧边栏底部提供设置入口，方便不通过托盘打开。

### 布局

参考 Swift `SettingsView`，采用左侧分类、右侧详情布局：

| 分类 | 内容 |
|---|---|
| 通用 | 日志保留天数、开机自启 |
| 项目 | 项目列表、添加/删除项目、服务显示隐藏、启动选中服务配置 |

整体保持当前暗色、紧凑、工具型风格。

### 通用设置

日志保留天数：

- 范围：`1...90`
- 默认：`7`
- 修改后持久化到 agent 级设置。
- agent 启动时按该值清理旧日志。

开机自启：

- 控制 SuperDev 桌面应用随系统登录启动。
- 启动后仍由 Tauri 正常拉起 agent。
- 不引入 agent 独立后台常驻模式。
- 使用 Tauri 插件或 Tauri 命令封装系统登录项状态。

### 项目设置

项目列表展示：

- 项目名。
- 根目录。
- 服务数量。

项目操作：

- 添加项目：复用当前目录选择逻辑和 `POST /api/projects`。
- 删除项目：走现有 `DELETE /api/projects/{id}`。

服务显示/隐藏：

- 控制菜单栏 popover 是否展示该服务。
- 存储在桌面端本地持久化状态中。
- 不写入项目 `.superdev/config.yaml`。

启动选中服务配置：

- 复用现有 `PUT /api/projects/{id}/selected`。
- 必选服务始终选中且不能取消。
- 非必选服务可勾选或取消。

重新加载配置：

- 本轮不实现重新加载配置按钮。
- 等 agent 有明确 reload 接口和行为约定后，再单独设计这个入口。

## 后端设置 API

日志保留天数需要 agent 级设置能力。新增轻量设置接口：

```http
GET /api/settings
PUT /api/settings
```

响应和请求体：

```json
{
  "log_retention_days": 7
}
```

规则：

- `log_retention_days` 范围为 1 到 90。
- 缺省值为 7。
- 设置文件存放在 agent 数据目录，和 `logs.db`、`projects.json` 同级。
- agent 启动时加载设置，并调用已有 `store.DeleteOlderThan(days)` 清理旧日志。

---

## 前端状态边界

| 状态 | 存储位置 | 说明 |
|---|---|---|
| 最近历史日志恢复 | `logStore` 内存 buffer | 从 `GET /api/logs` 拉取后合并到实时日志 |
| 历史/实时分隔线 | 前端展示状态 | 不写入后端，不是 `LogEntry` |
| 项目过滤规则 | 项目 `.superdev/config.yaml` | 通过 agent rules API 持久化 |
| 临时过滤 chip | `filterStore.panelFilters` | 当前面板 UI 状态 |
| 日志保留天数 | agent 设置文件 | 全局设置 |
| 开机自启 | 系统登录项 | 桌面应用级设置 |
| 服务显示/隐藏 | 桌面端本地持久化 | 只影响 popover 展示 |

## 风险与处理

1. WebSocket 打开时也会推送最近 200 条内存日志，前端需要按日志 ID 合并，避免历史恢复和 WS recent 重复显示。
2. 历史/实时分隔线不能混入真实日志数组，否则会影响过滤、书签、复制导出。
3. 规则保存是全量 `PUT`，弹层编辑期间需要基于最新规则列表保存，避免覆盖其它面板刚切换的 enabled 状态。
4. 开机自启是平台能力，Tauri 插件不可用时需要用 Tauri 命令封装，并在 UI 中展示错误状态。

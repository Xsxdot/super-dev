# 项目标签页与跨服务日志搜索设计

**日期**：2026-05-20  
**状态**：待用户审阅  
**范围**：`desktop` Tauri/Vue 客户端 + `agent` 日志搜索 API

---

## 背景

当前右侧日志区以“点击侧边栏服务后替换当前日志面板”为主，适合查看单个服务的实时输出，但不适合跨服务排障。用户定位 bug 时常见流程是输入一个 traceID、orderID 或错误关键字，从当前项目多个微服务的历史日志中重建一次调用链。

现有“历史记录”和“过滤规则”入口更像日志面板内的辅助按钮，无法承载跨服务搜索。新的设计把右侧区域升级为项目工作区：项目服务日志和搜索结果都以标签页形式打开，但共享同一个项目上下文。

---

## 目标

1. 右侧日志区顶部新增工作区标签页，项目成为主要上下文。
2. 点击侧边栏服务时，进入对应项目标签页并打开/聚焦该服务日志。
3. 侧边栏项目标题右侧新增搜索入口，每次点击都新开一个搜索标签页。
4. 搜索标签页默认搜索当前项目所有服务的历史日志，不再以 run 作为主要入口。
5. 搜索结果进入跨服务排障看板：左侧展示命中服务和命中时间线，右侧按服务分栏展示同一时间段上下文。
6. 点击左侧时间线日志时，右侧未固定服务统一跳转到对应时间段，加载目标日志前后的上下文，并高亮目标日志。

---

## 非目标

1. 不做跨项目搜索。一次搜索只属于一个项目。
2. 不把 run 作为用户主要概念。run 可作为底层字段保留，但 UI 不要求用户选择 run。
3. 不实现复杂查询语言。第一版只支持普通关键词/traceID 搜索。
4. 不做全文搜索索引。MVP 使用 SQLite `LIKE` 查询实现；数据量变大后再评估 FTS5。
5. 不改变现有日志采集和 WebSocket 实时推送机制。

---

## 信息架构

### 工作区标签类型

右侧主内容区顶部新增一层标签栏，标签分两类：

| 类型 | 示例标题 | 数量规则 | 用途 |
|---|---|---|---|
| 项目标签 | `super-debug` | 每个项目最多一个 | 查看该项目内服务实时/历史日志，保留现有分栏能力 |
| 搜索标签 | `Search: trace-8f21` | 每次点击搜索都新建 | 承载一次独立排障搜索上下文 |

项目标签和搜索标签都可以关闭。关闭后状态从前端内存移除；后续点击服务或搜索可重新创建。

### 标签切换规则

点击侧边栏服务时：

1. 如果该服务已经在某个项目标签的面板里打开，切换到该项目标签，并聚焦该服务所在面板。
2. 如果该服务未打开，但该项目标签已存在，切换到该项目标签，并在当前/默认面板打开该服务日志。
3. 如果该服务未打开，且该项目标签不存在，新建项目标签，再打开该服务日志。

点击侧边栏项目搜索按钮时：

1. 总是新建一个搜索标签。
2. 搜索标签绑定当前项目 ID。
3. 新标签打开后处于搜索空状态，输入框自动聚焦。

## 项目标签页

项目标签页内部复用现有日志面板体系：

- 保留 `PanelLayout`、多面板分栏、拖拽服务到面板、书签录制等能力。
- 项目标签维护自己的 panel layout，避免不同项目互相污染面板状态。
- 项目标签标题使用项目名；如果有多个项目标签，用户可以在顶部快速切换。
- 过滤规则管理仍属于项目上下文，可在项目标签内打开。

MVP 只保留当前会话内的标签状态，不要求跨应用重启恢复。

## 搜索标签页

### 空状态

搜索页初始状态类似搜索引擎：

- 页面中央显示项目名和搜索输入框。
- 输入框 placeholder 示例：`输入 traceID、orderID、错误关键字...`
- 回车或点击搜索按钮开始查询。
- 项目由搜索标签固定，不需要用户在页内再次选择项目。

### 搜索中

- 输入框保持可见。
- 主区域显示加载状态。
- 如果用户再次提交新关键词，取消或忽略上一轮过期响应，只展示最新关键词结果。

### 结果状态

结果页是一个跨服务排障看板，分为左右两区：

1. 顶部搜索栏：关键词、再次搜索入口、总命中数、时间范围。
2. 左侧导航区：上方是命中服务和数量，下方是命中日志时间线。
3. 右侧上下文区：按服务分栏展示同一时间段附近的日志内容。

左侧命中服务区：

- 位于左侧上方，最大高度约为左侧可视区域的三分之一。
- 内容可滚动。
- 每行显示服务颜色、服务名、命中数量、显示/隐藏开关。
- 隐藏服务后，左侧时间线和右侧分栏都立即移除该服务。

左侧时间线区：

- 位于命中服务区下方，占用左侧剩余空间。
- 只展示匹配搜索关键词的日志，用于快速选择调查锚点。
- 按 `timestamp ASC, id ASC` 排序。
- 点击任意命中日志，将该日志时间设为右侧上下文区的同步锚点。

每条时间线日志显示：

- 时间戳，精确到毫秒。
- 服务名和服务颜色。
- 日志级别。
- 消息内容，关键词高亮。
- 可选的 stream/run 信息作为次要元数据。

右侧服务分栏：

- 每个可见命中服务一列；服务列可以横向滚动。
- 每列顶部显示服务名、命中数量、固定/取消固定按钮。
- 默认所有未固定服务跟随左侧时间线选中的时间同步跳转。
- 某个服务定位到关键日志后，用户可以点击固定；固定后该服务列不再跟随左侧时间线跳转，直到取消固定。
- 每列展示该服务在当前时间锚点附近的上下文日志，而不只展示命中关键词的日志。

### 无结果

- 显示关键词和项目名。
- 提示当前项目没有匹配日志。
- 保留搜索框，允许直接修改关键词重新搜索。

### 失败

- 显示错误信息。
- 提供重试按钮。
- 不清空用户输入。

## 跨服务排障逻辑

搜索结果默认展示当前项目所有服务的命中日志。用户输入 traceID 后，系统先帮助判断“这是不是一条真实链路”：

1. 哪些服务命中了。
2. 每个服务命中多少条。
3. 命中日志按时间合并后是否形成连续序列。

如果某个服务只是碰巧命中同样关键字，用户可以在左侧命中服务区隐藏该服务。隐藏后：

- 时间线立即重排。
- 右侧服务分栏立即移除该服务列。
- 总命中数展示可见命中数和原始命中数。
- 隐藏状态只影响当前搜索标签，不写入项目过滤规则。

右侧服务分栏不是替代时间线，而是用于回答“同一时间附近各服务分别发生了什么”。左侧时间线负责选择调查锚点，右侧分栏负责对齐上下文。

### 时间对齐规则

右侧分栏使用时间栅格对齐，而不是要求日志毫秒级时间完全相同。

- MVP 使用 1 秒作为时间栅格，栅格标签显示为 `HH:mm:ss`。
- 每个栅格收集所有可见服务在该秒内的上下文日志。
- 如果某个时间栅格内所有服务都没有日志，则不渲染该栅格，不产生空白。
- 如果至少一个服务在该栅格有日志，而某个服务没有日志，该服务列显示一行空白占位。
- 同一时间栅格内，各服务列的行高取该栅格中最高内容高度，保证横向对齐。
- 用户点击左侧时间线日志时，右侧未固定服务加载该日志时间前后的上下文，并滚动到包含该日志的时间栅格。
- 固定服务列保持自身当前锚点和滚动位置，不参与新的时间同步。

这套规则对应排障直觉：时间上没有任何服务活动时不制造空白；只要某些服务在同一时间附近有活动，其他服务的空白占位能保留横向时间对齐。

## 后端 API

### 新增搜索接口

```http
GET /api/log-search?project={projectId}&q={keyword}&limit={n}
```

可选参数：

```http
service={serviceId}        # 可重复，限制服务集合
from={isoTimestamp}        # 可选时间下界
to={isoTimestamp}          # 可选时间上界
before={id}                # 可选分页游标
```

响应：

```json
{
  "query": "trace-8f21",
  "total": 42,
  "items": [
    {
      "id": 123,
      "service_id": "svc-api",
      "run_id": "run-1",
      "timestamp": "2026-05-20T12:31:02.014Z",
      "level": "INFO",
      "message": "request accepted trace-8f21",
      "stream": "stdout"
    }
  ],
  "service_counts": {
    "svc-api": 12,
    "svc-worker": 5,
    "svc-billing": 3
  }
}
```

### 查询语义

- `project` 必填。agent 根据内存项目列表找到该项目的 service IDs。
- `q` 必填，去除首尾空白后不能为空。
- MVP 仅对 `message` 做大小写不敏感的包含匹配。
- 默认按 `timestamp ASC, id ASC` 返回，便于重建时间线。
- `limit` 默认 1000，最大 5000。
- 如果需要分页，下一页继续按时间线追加。

### 新增上下文接口

```http
GET /api/logs/context?id={logId}&project={projectId}&before_ms={n}&after_ms={n}
```

响应：

```json
{
  "target_id": 123,
  "anchor_time": "2026-05-20T12:31:02.014Z",
  "items_by_service": {
    "svc-api": [
      {
        "id": 120,
        "service_id": "svc-api",
        "run_id": "run-1",
        "timestamp": "2026-05-20T12:31:01.880Z",
        "level": "INFO",
        "message": "previous log",
        "stream": "stdout"
      }
    ],
    "svc-worker": []
  }
}
```

查询语义：

- `id` 必填。agent 先找到目标日志的 `timestamp`，作为 `anchor_time`。
- `project` 必填。agent 根据项目服务集合查询上下文。
- `service` 可重复传入，用于限制上下文服务集合；不传则使用项目全部服务。
- `before_ms` 默认 30000，`after_ms` 默认 30000，最大各 300000。
- 返回各服务在 `[anchor_time - before_ms, anchor_time + after_ms]` 范围内的日志。
- 每个服务内部按 `timestamp ASC, id ASC` 排序。
- 前端用 `target_id` 高亮目标日志，并按 1 秒时间栅格把 `items_by_service` 渲染为同步分栏。

### 实现边界

- API 层只负责解析参数、校验 project/service 范围、返回 JSON。
- store 层新增搜索方法，封装 SQL 查询。
- 不在 agent 做项目级 LogRule 过滤。搜索应返回原始历史日志，前端只做当前搜索标签内的服务隐藏。

## 前端状态模型

新增 workspace store，负责标签页：

```ts
type WorkspaceTab =
  | ProjectWorkspaceTab
  | SearchWorkspaceTab

interface ProjectWorkspaceTab {
  id: string
  type: 'project'
  projectId: string
  title: string
  layoutRoot: PanelNode
  focusedPanelId: string | null
}

interface SearchWorkspaceTab {
  id: string
  type: 'search'
  projectId: string
  title: string
  query: string
  status: 'empty' | 'loading' | 'results' | 'emptyResults' | 'error'
  results: LogEntry[]
  serviceCounts: Record<string, number>
  hiddenServiceIds: string[]
  selectedLogId: number | null
  contextAnchorTime: string | null
  contextByService: Record<string, LogEntry[]>
  pinnedServiceIds: string[]
  error: string | null
}
```

现有 `panelStore` 可以逐步迁移：

- 第一阶段：保留 `panelStore`，先让 workspace project tab 持有当前 root。
- 后续阶段：让每个 project tab 拥有独立 layout root。

为了避免一次改动过大，实施时应优先完成第一阶段行为闭环，再拆分多项目布局状态。

## 入口交互

### 侧边栏服务点击

服务行点击不再简单替换全局当前面板，而是调用 workspace store：

```ts
openService(projectId, serviceId)
```

该方法内部按“已有服务面板 → 已有项目标签 → 新建项目标签”的规则处理。

### 侧边栏项目搜索

项目标题右侧新增搜索图标按钮：

```ts
openSearch(projectId)
```

每次调用都创建新的 search tab，并切换过去。

按钮应位于“启动选中/停止”旁边，图标优先使用现有图标体系；若当前项目没有服务，可禁用。

## 结果跳转

点击搜索结果时，前端执行：

1. 根据 `service_id` 找到项目和服务。
2. 在当前搜索标签内设置 `selectedLogId`。
3. 调用 `/api/logs/context?id={logId}&project={projectId}` 加载项目内可见服务的同时间上下文。
4. 右侧所有未固定服务列同步到返回的 `anchor_time`。
5. 固定服务列保持原上下文和滚动位置。
6. 右侧按 1 秒时间栅格渲染上下文，并高亮目标日志。

如果用户需要回到普通服务日志面板，可以在右侧服务列中点击“打开服务日志”，再调用 `openService(projectId, serviceId)` 切换到项目标签。

---

## 与现有过滤规则的关系

跨服务搜索不是过滤规则的替代品。

- 过滤规则：项目级持久化，用来长期降低噪声，例如隐藏 heartbeat。
- 搜索标签：一次性排障上下文，用来从历史日志中找证据。
- 搜索里的服务隐藏：只影响当前搜索标签，不持久化。

现有“过滤规则管理”仍保留在项目标签的日志工具栏内，由单独任务补完整弹窗。

### agent

- store 搜索方法：按项目服务集合、关键词、时间排序、limit、大小写不敏感匹配。

## 设计决策

1. **项目标签而不是服务标签**：服务数量多时，服务级 tab 会快速爆炸；项目才是用户排障的自然上下文。
2. **搜索每次新开 tab**：一次搜索代表一次调查，多个关键词需要并存对照，不能互相覆盖。
3. **左侧时间线 + 右侧服务分栏**：时间线用于选择调查锚点，服务分栏用于对比同一时间附近各服务的上下文。
4. **服务隐藏是局部状态**：隐藏“碰巧命中”的服务是当前调查动作，不应污染项目过滤规则。
5. **不以 run 作为入口**：用户通常知道 traceID，不知道 run；run 是底层分区字段，不应成为排障入口。
